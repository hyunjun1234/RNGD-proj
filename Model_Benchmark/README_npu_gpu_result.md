# NPU vs GPU 벤치마크 결과 비교 — Furiosa RNGD vs NVIDIA A6000

RNGD NPU(`furiosa-llm`)와 NVIDIA A6000 GPU(`vLLM`)에서 **같은 모델·같은 조건**으로
LLM 서빙을 측정해 직접 비교한 문서. 측정 자동화·원본 데이터는 [`rngd-npu/`](rngd-npu/README.md)·[`bench-gpu/`](bench-gpu/README.md) 참조.

> 측정일: 2026-05. NPU = RNGD 1카드(8 PE) · GPU = RTX A6000 48GB 1장.
> 원본 리포트: [`rngd-npu/REPORT.md`](rngd-npu/REPORT.md) · [`bench-gpu/REPORT.md`](bench-gpu/REPORT.md)

## 1. TL;DR — 3줄 결론

1. **8B 모델 단일 사용자 속도는 NPU가 빠르다.** Llama-3.1-8B 단일 출력 속도 NPU 54.5 vs GPU 41.9 tok/s — **NPU가 30% 우세**. 토큰 간 지연(ITL)도 NPU가 23% 짧다. RNGD는 **지연(latency) 최적화 디바이스**다.
2. **동시 사용자가 많아지면 GPU가 역전한다.** Llama 기준 동시성 c16까지는 NPU 우세, **c32 부근에서 교차**, c128에서는 GPU가 66% 앞선다. GPU는 **처리량(throughput) 최적화 디바이스**다.
3. **정확도(SWE-bench)는 두 디바이스 모두 0%** — 코드 수정 능력은 하드웨어가 아니라 모델 크기가 결정한다. 8B single-shot으로는 NPU든 GPU든 SWE-bench Lite를 못 푼다.

| 한눈 지표 (Llama-3.1-8B) | NPU (RNGD) | GPU (A6000) | 승자 |
|---|--:|--:|:--:|
| 단일 출력 속도 (tok/s) | **54.5** | 41.9 | 🟦 NPU +30% |
| 토큰 간 지연 ITL p50 (ms) | **18.3** | 23.9 | 🟦 NPU |
| 첫 토큰 지연 TTFT p50 (ms) | **32.6** | 35.9 | 🟦 NPU |
| 저동시성 합산 처리량 (c8) | **335** | 272 | 🟦 NPU |
| 고동시성 합산 처리량 (c128) | 2,192 | **3,634** | 🟥 GPU +66% |
| SWE-bench resolved | 0/50 | 0/47 | ⬜ 무승부 |

## 2. 비교가 공정한 이유 — 조건 정렬

NPU와 GPU는 서빙 엔진이 다르다(`furiosa-llm` vs `vLLM`). 그래서 **엔진을 제외한 모든 변수를 일치**시켰다. 특히 처음 GPU 측정은 컨텍스트 윈도가 8배 차이(GPU 32K vs NPU 4K) 나서 불공정했고, 이를 다시 맞춰 재측정했다.

| 정렬 항목 | NPU (RNGD) | GPU (A6000) | 일치? |
|---|---|---|:--:|
| 측정 모델 | `furiosa-ai/Qwen2.5-0.5B`·`Llama-3.1-8B` | `Qwen/Qwen2.5-0.5B`·`meta-llama/Llama-3.1-8B` | ✅ 동일 모델 |
| 정밀도 | bf16 (양자화 없음) | bf16 (양자화 없음) | ✅ |
| 컨텍스트 — Qwen | 4,096 (artifact 컴파일 한도) | 4,096 (`--max-model-len 4096`) | ✅ 맞춤 |
| 컨텍스트 — Llama | 32,768 (artifact 한도) | 32,768 (`--max-model-len 32768`) | ✅ 맞춤 |
| prefix caching | ON | ON (`--enable-prefix-caching`) | ✅ |
| sweep 그리드 | 동시성 1–128 × prompt 256/1024/4096 | 동일 | ✅ |
| SWE-bench | Lite oracle · single-shot · 50건 stratified | 동일 | ✅ |
| 측정 클라이언트 | 동일 코드 (OpenAI 호환 API) | 동일 코드 | ✅ |
| 디바이스 수 | RNGD 1카드 | A6000 1장 | ✅ 단일 디바이스 |

**검증 포인트**: 컨텍스트 정렬이 실제로 됐는지는 sweep `prompt=4096` 결과로 확인된다 —
**NPU Qwen·GPU Qwen 모두 prompt=4096에서 전건 실패**(`prompt 4096 + max_tokens 256 > 4096` 한도 초과).
양쪽이 똑같이 실패했다는 것이 4K 조건이 동일하게 적용됐다는 증거다.

> ⚠️ 이 비교는 **성능 비교**다. A6000과 RNGD는 가격·소비전력(TDP)·폼팩터가 다른 제품군이므로,
> 전력당 성능·비용당 성능(TCO)은 별도 측정이 필요하다. 이 문서는 "동일 SW 조건에서의 처리 성능"만 다룬다.

## 3. 테스트 1 — 단일 요청 토큰 속도

동시 사용자 1명일 때의 체감 속도. 코드 자동완성·대화형 보조의 핵심 지표.

### Llama-3.1-8B-Instruct (코드 생성 후보 모델)

| 지표 | NPU (RNGD) | GPU (A6000) | 해석 |
|---|--:|--:|---|
| 출력 TPS p50 (tok/s) | **54.5** | 41.9 | NPU가 **1.30배** 빠름 |
| TTFT p50 (첫 토큰, ms) | **32.6** | 35.9 | NPU가 약간 빠름 |
| TTFT p95 (ms) | **37.0** | 41.1 | NPU가 꼬리 지연도 짧음 |
| ITL p50 (토큰 간격, ms) | **18.3** | 23.9 | NPU가 토큰을 23% 더 촘촘히 |

→ **8B급 모델에서 RNGD는 GPU보다 한 사람에게 더 빠르게 응답한다.** 토큰 생성이 GPU 대비 30% 빠르고,
첫 토큰도 빠르게 나온다. 챗봇·IDE 자동완성처럼 "한 사람의 응답 속도"가 중요한 워크로드에서 NPU가 유리하다.

### Qwen2.5-0.5B-Instruct (smoke·검증용 모델)

| 지표 | NPU (RNGD) | GPU (A6000) | 해석 |
|---|--:|--:|---|
| 출력 TPS p50 (tok/s) | 84.5 | **249.0** | GPU가 **2.95배** 빠름 |
| TTFT p50 (ms) | 30.6 | **13.5** | GPU가 2.3배 빠름 |
| ITL p50 (ms) | 11.9 | **4.0** | GPU가 3배 빠름 |

→ **0.5B처럼 아주 작은 모델은 GPU가 압도한다.** 모델이 너무 작아 연산보다 메모리 대역폭이
병목이고, A6000의 높은 대역폭이 그대로 속도로 나타난다. 단 Qwen-0.5B는 파이프라인 검증용
**smoke 모델**이라 실사용 후보가 아니다 — 이 수치는 "작은 모델에서는 GPU가 유리"라는 참고용.

> **왜 8B에선 NPU, 0.5B에선 GPU인가?** RNGD는 트랜스포머 추론에 특화된 NPU로,
> 모델이 일정 규모(수 B 파라미터) 이상일 때 연산 파이프라인 효율이 살아난다.
> 0.5B는 그 효율이 발휘되기엔 너무 작아 메모리 대역폭 싸움이 되고, 거기선 GPU가 앞선다.

## 4. 테스트 2 — 동시성 스케일링 (가장 중요한 그래프)

동시 요청을 1→128로 늘리며 **합산 처리량**(서버 전체가 초당 뱉는 토큰)을 측정. prompt_len=1024 기준.

### Llama-3.1-8B — 합산 처리량 (tok/s)

| 동시성 | NPU (RNGD) | GPU (A6000) | 우세 | 격차 |
|--:|--:|--:|:--:|--:|
| 1 | **52.4** | 41.3 | 🟦 NPU | +27% |
| 2 | **99.8** | 79.5 | 🟦 NPU | +26% |
| 4 | **191.6** | 151.3 | 🟦 NPU | +27% |
| 8 | **335.1** | 271.9 | 🟦 NPU | +23% |
| 16 | **636.7** | 579.5 | 🟦 NPU | +10% |
| 32 | 1,091 | **1,122** | 🟥 GPU | +3% ← **교차점** |
| 64 | 1,647 | **2,048** | 🟥 GPU | +24% |
| 128 | 2,192 | **3,634** | 🟥 GPU | +66% |

```
합산 TPS (Llama-3.1-8B, prompt 1024)
4000 |                                            ●GPU 3634
3000 |                                       
2000 |                            ▲NPU 2192      ●GPU 2048
1000 |              ▲1091●1122                ▲NPU 1647
 500 |        ▲637●580
   0 |▲52●41___________________________________________
      c1    c8   c16   c32      c64        c128
      └── NPU 우세 ──┘ 교차 └──── GPU 우세 ────┘
```

**핵심**: Llama-3.1-8B는 **동시성 16명까지 NPU가 빠르고, 32명 부근에서 뒤집혀 128명에선 GPU가 66% 앞선다.**

- **NPU가 이기는 구간 (c1–c16)**: 소수 사용자가 쓰는 사내 코드 어시스턴트, IDE 플러그인.
- **GPU가 이기는 구간 (c32–c128)**: 대규모 동시 트래픽을 받는 공용 API 서버.
- 두 디바이스 모두 c1–c128 **전 구간 요청 실패 0건** — 안정성은 동등.

### Qwen2.5-0.5B — 합산 처리량 (tok/s)

| 동시성 | NPU (RNGD) | GPU (A6000) | 우세 |
|--:|--:|--:|:--:|
| 1 | 70.6 | **244.0** | 🟥 GPU |
| 8 | 480.8 | **1,481.9** | 🟥 GPU |
| 32 | 1,932 | **5,535** | 🟥 GPU |
| 128 | 4,120 | **11,423** | 🟥 GPU |

→ 0.5B는 **전 동시성 구간에서 GPU 우세** (peak 기준 2.8배). 작은 모델은 동시성을 올려도
NPU의 연산 파이프라인 이점이 살아나지 않는다.

> **요약**: 동시성 그래프가 NPU vs GPU 성격을 가장 잘 보여준다.
> RNGD = 낮은 동시성에서 빠른 **latency 머신**, A6000 = 높은 동시성에서 강한 **throughput 머신**.

## 5. 테스트 3 — serve 옵션 튜닝(memsweep)

서버 기동 옵션을 한 축씩 바꿔(OFAT) 합산 처리량 변화를 측정. 동시성 128 고정.

### 정상 범위 옵션 — 둘 다 튜닝 효과 작음

| 모델 | NPU baseline | NPU best | GPU baseline | GPU best |
|---|--:|--:|--:|--:|
| Llama-3.1-8B | 2,251 | 2,288 (`max_num_batched_tokens 4096`) | 3,616 | 3,671 (`max_model_len 4096`) |
| Qwen2.5-0.5B | 4,246 | 4,570 (`max_num_batched_tokens 16384`) | 11,854 | 11,854 (`baseline`) |

→ `max_model_len`·`max_num_batched_tokens`를 바꿔도 처리량 변동은 양쪽 모두 **수 % 이내**.
**기본값으로 서빙해도 손해가 거의 없다** — 운영 튜닝 부담이 적다는 뜻.

### 배치 슬롯 축소 — 결정적 차이 발견

`max_batch_size`(NPU) ↔ `max_num_seqs`(GPU)를 8로 줄였을 때:

| 모델 | NPU `max_batch_size=8` | GPU `max_num_seqs=8` |
|---|--:|--:|
| Llama-3.1-8B | 2,232 (baseline 대비 **−1%**) | 306 (baseline 대비 **−92%**) |
| Qwen2.5-0.5B | 4,073 (baseline 대비 **−4%**) | 1,649 (baseline 대비 **−86%**) |

→ **GPU(vLLM)는 동시 시퀀스 슬롯을 줄이면 처리량이 붕괴**한다(연속 배칭이 슬롯 수에 직접 의존).
**NPU(furiosa-llm)는 거의 영향 없음** — 배치 제어 방식이 근본적으로 다르다.
운영 시사점: GPU는 `max_num_seqs`를 충분히 크게 둬야 하고, NPU는 이 옵션에 민감하지 않다.

## 6. 테스트 4 — SWE-bench 코드 수정 정확도

실제 GitHub 이슈를 unified diff로 해결 → Docker harness로 실제 테스트 실행해 채점.
SWE-bench Lite oracle, single-shot, repo별 stratified 50건. (벤치마크 작동 원리는
[`bench-gpu/README.md §7`](bench-gpu/README.md#7-swe-bench-lite-깊이-분석--llm을-어떻게-채점하나) 참조)

| 모델 · 디바이스 | resolved | unresolved | 적용실패 | 컨텍스트 제외 | 평가된 건수 |
|---|--:|--:|--:|--:|--:|
| Llama-3.1-8B · **NPU** | 0 | 14 | 36 | 0 | 50 |
| Llama-3.1-8B · **GPU** | 0 | 18 | 29 | 3 | 47 |
| Qwen2.5-0.5B · **NPU** | 0 | 1 | 2 | 47* | 3 |
| Qwen2.5-0.5B · **GPU** | 0 | 1 | 2 | 47 | 3 |

\* NPU Qwen은 `/v1/models`가 컨텍스트 한도를 노출하지 않아 47건이 사전 필터 대신 추론 단계 에러(빈 패치)로 집계됐다 — 실질은 GPU와 동일하게 "4K 초과 47건 평가 불가".

### 분석

- **resolved는 4개 조합 모두 0** — 8B·0.5B를 single-shot으로 돌리면 어느 하드웨어에서도 SWE-bench Lite를 못 푼다. **정확도는 모델 크기·방식이 결정하지 디바이스가 결정하지 않는다.**
- **적용실패율**: NPU Llama 36/50(72%), GPU Llama 29/47(62%) — 둘 다 60~70%대로, 모델이 **문법적으로 올바른 unified diff를 잘 못 만든다**. (diff의 `@@` hunk 헤더 줄 수 불일치가 주원인)
- **Qwen은 NPU·GPU 똑같이 3건만 평가** — 4K 컨텍스트로는 SWE-bench oracle 프롬프트(평균 5K~30K 토큰)가 대부분 안 들어간다. 조건 정렬이 정확히 동작했음을 보여준다.
- **공개 GPU 수치 대조**: 같은 single-shot 방식이면 GPT-4(1.7%)·Claude 2(4.8%)·70B(5.4%)도 한 자릿수 — 8B는 그 아래. 정확도를 올리려면 **더 큰 모델(32B/70B)이나 에이전트 스캐폴드**가 필요하지, 하드웨어 교체로는 안 오른다.

> **이 테스트의 결론**: SWE-bench는 "NPU vs GPU" 승부가 아니라 "모델 능력의 하한"을 보여주는 지표.
> NPU/GPU 비교 관점에서는 **정확도가 디바이스 독립적**이라는 점이 핵심 — 즉 NPU로 옮겨도 정확도 손해가 없다.

## 7. 테스트 5 — 임베딩 / 리랭커

| 모델 · 디바이스 | batch=1 | batch=16 | batch=64 |
|---|--:|--:|--:|
| Qwen3-Embedding-8B · NPU | 1.17 inputs/s | 1.17 | 1.17 |
| Qwen3-Reranker-8B · NPU | 1.17 pairs/s | — | — |
| Qwen3-Embedding-8B · GPU | (미측정) | — | — |
| Qwen3-Reranker-8B · GPU | (미측정) | — | — |

- **NPU 임베딩/리랭커는 batch 크기와 무관하게 1.17/s 고정** — 항목당 약 0.85초. 8B 생성 모델의 단일 prefill(수십 ms) 대비 수십 배 느리다.
- 원인: prebuilt artifact가 **단일 배치(batch=1)로 컴파일**된 것으로 추정 — 서버 스케줄러가 요청을 1개씩 NPU에 던진다(`npu_queue_limit: 1`).
- **GPU 측은 미측정** — `bench-gpu`에서 embedding/reranker 모델을 `enabled: false`로 둔 상태(vLLM 0.21의 `--runner pooling` 셋업 필요). 직접 비교 불가.

> **시사점**: 현재 NPU 임베딩 artifact는 대량 검색(RAG)에 부적합. 실사용 전 배칭 지원 artifact 재컴파일 또는 GPU 측 임베딩 측정으로 보완 필요. 이 항목은 후속 과제.

## 8. 종합 — 어떤 워크로드에 무엇을 쓰나

### 측정으로 확인된 디바이스 성격

| | RNGD NPU | A6000 GPU |
|---|---|---|
| 강점 | **단일/저동시성 latency** (8B 단일 +30%) | **고동시성 throughput** (c128 +66%) |
| 약점 | 고동시성 처리량, 작은 모델 | 저동시성에서 8B 단일 속도 |
| serve 옵션 민감도 | 낮음 (배치 슬롯에 둔감) | 높음 (`max_num_seqs` 필수 튜닝) |
| 코드 정확도(SWE-bench) | 0% (모델 한계) | 0% (모델 한계) — **디바이스 무관** |

### 워크로드별 권장

| 워크로드 | 권장 | 근거 |
|---|---|---|

### 한 줄 결론

> **8B급 코드 생성 모델을 소수~중간 규모(동시 ≤16~32)로 서빙한다면 RNGD NPU가 GPU보다
> 빠르고 안정적이다.** 동시 사용자가 수십 명을 넘는 대규모 트래픽이라면 GPU의 배치 처리량이
> 앞선다. 코드 수정 정확도는 두 디바이스가 동일(모델이 결정) — NPU 전환에 따른 정확도 손해는 없다.

## 9. 측정의 한계

- **모델 규모**: 실측 가능한 코드 생성 모델은 Llama-3.1-8B 단독. 강력 후보인 32B·70B는 NPU prebuilt artifact가 `tensor_parallel=32`(RNGD 4카드)로 컴파일돼 현 2카드 환경에서 서빙 불가 → 제외. 32B/70B 비교는 4카드 환경 측정이 필요하다.
- **전력·비용**: 이 문서는 성능만 다룬다. RNGD와 A6000은 TDP·가격대가 다르므로 전력당/비용당 성능(TCO)은 별도 측정 과제.
- **임베딩/리랭커**: GPU 측 미측정 → embedding/reranker는 NPU 단독 수치만 존재.
- **SWE-bench 표본**: Lite 300건 중 50건 stratified subset. 전체 300건·Verified 500건으로 확대하면 수치가 달라질 수 있으나, resolved 0%라는 큰 그림은 바뀌지 않을 가능성이 높다.
- **GPU 결과 경로**: `bench-gpu/REPORT.md`의 결과 경로(`/home/etri/...`)는 A6000 서버에서 생성된 값. 로컬에서 `report.py` 재실행 시 갱신된다.
