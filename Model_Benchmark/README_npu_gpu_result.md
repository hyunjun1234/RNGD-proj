# RNGD NPU와 A6000 GPU 벤치마크 결과 비교

Furiosa RNGD NPU와 NVIDIA A6000 GPU에서 같은 모델을 같은 조건으로 측정해 비교한 문서입니다.
측정에 쓴 코드와 원본 데이터는 [rngd-npu](rngd-npu/README.md), [bench-gpu](bench-gpu/README.md) 폴더에 있습니다.

측정 시점은 2026년 5월이고, NPU는 RNGD 1카드(8 PE), GPU는 RTX A6000 48GB 1장을 사용했습니다.

## 요약

코드 생성 후보 모델인 Llama-3.1-8B를 기준으로 결과를 정리하면 이렇습니다.

사용자 한 명이 쓸 때는 NPU가 빠릅니다. 초당 생성하는 토큰이 NPU 54.5개, GPU 41.9개로 30%쯤 차이가 납니다.
하지만 동시 사용자가 늘어나면 상황이 달라집니다. 동시 16명까지는 NPU가 앞서다가 32명 근처에서 GPU가 따라잡고,
128명에서는 GPU가 66% 앞섭니다.

코드 수정 정확도는 두 장비 모두 0%로 같았습니다. 정확도는 어떤 장비를 쓰는지가 아니라 모델 크기에 달려 있어서,
8B 모델로는 NPU든 GPU든 SWE-bench 문제를 풀지 못했습니다.

| Llama-3.1-8B 기준 | NPU (RNGD) | GPU (A6000) | 우세 |
|---|--:|--:|---|
| 단일 사용자 출력 속도 | 54.5 tok/s | 41.9 tok/s | NPU 30% 빠름 |
| 첫 토큰까지 걸린 시간 | 32.6 ms | 35.9 ms | NPU |
| 토큰 사이 간격 | 18.3 ms | 23.9 ms | NPU |
| 동시 8명 합산 처리량 | 335 tok/s | 272 tok/s | NPU |
| 동시 128명 합산 처리량 | 2,192 tok/s | 3,634 tok/s | GPU 66% 빠름 |
| SWE-bench 해결 건수 | 0 / 50 | 0 / 47 | 둘 다 0 |

한 줄로 정리하면, RNGD는 응답 속도에 강한 장비이고 A6000은 많은 요청을 한꺼번에 처리하는 데 강한 장비입니다.

## 측정 조건을 맞춘 방법

NPU와 GPU는 서빙 엔진이 다릅니다. NPU는 furiosa-llm, GPU는 vLLM을 씁니다. 그래서 엔진을 빼고 나머지 조건은
모두 똑같이 맞췄습니다.

처음 GPU를 측정했을 때는 컨텍스트 길이가 NPU의 8배(32K 대 4K)여서 비교가 공정하지 않았습니다.
NPU에서 쓰는 prebuilt 모델은 컴파일할 때 컨텍스트 길이가 고정되는데, Qwen2.5-0.5B는 4,096 토큰,
Llama-3.1-8B는 32,768 토큰입니다. 그래서 GPU 쪽도 `--max-model-len` 옵션으로 같은 값에 맞춰 다시 측정했고,
이 문서의 데이터는 그렇게 다시 측정한 결과입니다.

| 항목 | NPU (RNGD) | GPU (A6000) |
|---|---|---|
| 측정 모델 | Qwen2.5-0.5B, Llama-3.1-8B | 같은 모델 (원본 HuggingFace) |
| 정밀도 | bf16 | bf16 |
| 컨텍스트 길이 (Qwen) | 4,096 | 4,096 |
| 컨텍스트 길이 (Llama) | 32,768 | 32,768 |
| prefix caching | 켬 | 켬 |
| 동시성 측정 범위 | 1~128명 | 1~128명 |
| SWE-bench | Lite oracle 50건 | 같음 |

조건이 실제로 맞았는지는 다음으로 확인할 수 있습니다. 동시성 측정에서 프롬프트 길이를 4,096으로 잡은 항목은
NPU와 GPU 모두 전부 실패했습니다. 프롬프트 4,096에 응답 256을 더하면 4,096 한도를 넘기 때문인데,
양쪽이 똑같이 실패했다는 건 4K 제한이 동일하게 적용됐다는 뜻입니다.

한 가지 짚어둘 점은, 이 비교가 성능만 다룬다는 것입니다. A6000과 RNGD는 가격대와 소비전력이 다른 제품이라,
전력당 성능이나 비용 대비 성능까지 보려면 따로 측정해야 합니다.

## 테스트 1: 단일 요청 속도

사용자 한 명이 요청을 보냈을 때 얼마나 빨리 답하는지를 봅니다. 코드 자동완성이나 대화형 보조처럼
한 사람의 체감 속도가 중요한 작업에서 의미 있는 지표입니다.

Llama-3.1-8B (코드 생성 후보 모델):

| 지표 | NPU | GPU | 우세 |
|---|--:|--:|---|
| 출력 속도 | 54.5 tok/s | 41.9 tok/s | NPU 30% 빠름 |
| 첫 토큰까지 시간 | 32.6 ms | 35.9 ms | NPU |
| 토큰 사이 간격 | 18.3 ms | 23.9 ms | NPU |

8B 모델에서는 NPU가 전반적으로 빠릅니다. 토큰 생성 속도가 30% 빠르고, 첫 토큰이 나오는 시점이나
토큰 사이 간격도 NPU가 더 짧습니다. 챗봇이나 IDE 자동완성처럼 한 사람의 응답 속도가 중요한 경우에 NPU가 유리합니다.

Qwen2.5-0.5B (동작 확인용 모델):

| 지표 | NPU | GPU | 우세 |
|---|--:|--:|---|
| 출력 속도 | 84.5 tok/s | 249.0 tok/s | GPU 약 3배 |
| 첫 토큰까지 시간 | 30.6 ms | 13.5 ms | GPU |

반대로 0.5B처럼 아주 작은 모델은 GPU가 3배 가까이 빠릅니다. 모델이 너무 작으면 연산량보다
메모리에서 데이터를 읽어오는 속도가 병목이 되는데, 이 부분은 A6000이 유리합니다. 다만 Qwen2.5-0.5B는
파이프라인이 제대로 도는지 확인하려고 넣은 모델이라 실제 후보는 아닙니다.

RNGD는 어느 정도 규모가 되는 모델에서 연산 효율이 살아나는 구조입니다. 8B에서는 그 효과가 나타나 NPU가 앞서고,
0.5B는 너무 작아 그 효과가 나타나지 않습니다.

## 테스트 2: 동시 사용자 처리

동시 요청 수를 1명에서 128명까지 늘리면서 서버 전체가 초당 만들어내는 토큰 수, 즉 합산 처리량을 측정했습니다.
프롬프트 길이는 1,024 기준입니다.

Llama-3.1-8B 합산 처리량 (tok/s):

| 동시 사용자 | NPU | GPU | 우세 |
|--:|--:|--:|---|
| 1 | 52 | 41 | NPU |
| 8 | 335 | 272 | NPU |
| 16 | 637 | 580 | NPU |
| 32 | 1,091 | 1,122 | 비슷함 |
| 64 | 1,647 | 2,048 | GPU |
| 128 | 2,192 | 3,634 | GPU |

동시 16명까지는 NPU가 앞섭니다. 32명 근처에서 두 장비가 비슷해지고(1,091 대 1,122), 그 뒤로는 GPU가 점점 벌립니다.
128명에서는 GPU가 66% 앞섭니다.

정리하면, 소수의 사용자가 쓰는 환경(사내 코드 도구, IDE 플러그인)이면 NPU가 낫고, 동시 접속이 수십 명을 넘는
공용 API 서버라면 GPU가 낫습니다. 두 장비 모두 1명부터 128명까지 요청 실패는 한 건도 없었습니다.

Qwen2.5-0.5B는 전 구간에서 GPU가 앞섰습니다. 128명 기준 NPU 4,120, GPU 11,423 tok/s로 GPU가 2.8배입니다.
작은 모델은 동시성을 올려도 NPU 쪽 이점이 살아나지 않습니다.

## 테스트 3: 서버 옵션 튜닝

서버를 띄울 때 주는 옵션을 하나씩 바꿔가며 처리량이 얼마나 달라지는지 봤습니다. 동시성 128명 기준입니다.

`max-model-len`이나 `max-num-batched-tokens` 같은 옵션을 바꿔봐도 처리량 변화는 NPU, GPU 모두 몇 퍼센트 안에
머물렀습니다. 기본값으로 서빙해도 큰 손해가 없다는 뜻이라, 운영할 때 옵션 튜닝에 신경 쓸 필요가 적습니다.

차이가 두드러진 건 동시에 처리할 수 있는 요청 슬롯 수를 줄였을 때입니다. NPU는 `max_batch_size`,
GPU는 `max_num_seqs`라는 옵션이 이에 해당합니다.

| Llama-3.1-8B | 슬롯 8로 줄였을 때 합산 처리량 |
|---|--:|
| NPU | 2,232 (기본값과 거의 같음) |
| GPU | 306 (기본값의 8% 수준) |

GPU는 슬롯을 8로 줄이면 처리량이 92% 무너집니다. vLLM의 배치 처리가 슬롯 수에 직접 묶여 있기 때문입니다.
NPU는 같은 조건에서 거의 영향이 없습니다. 운영할 때 GPU는 이 값을 충분히 크게 잡아야 하고,
NPU는 크게 신경 쓰지 않아도 됩니다.

## 테스트 4: 코드 수정 정확도 (SWE-bench)

SWE-bench는 실제 GitHub 이슈를 모델이 코드 패치로 해결하게 하고, 그 패치를 진짜 테스트로 돌려
통과 여부를 채점하는 평가입니다. Lite 데이터셋의 50건을, 고칠 파일을 미리 알려주는 oracle 방식으로,
한 번에 답을 받는 방식으로 측정했습니다. 작동 방식은 [bench-gpu/README.md](bench-gpu/README.md)에 자세히 정리해 두었습니다.

| 모델 · 장비 | 해결 | 미해결 | 적용 실패 | 채점한 건수 |
|---|--:|--:|--:|--:|
| Llama-3.1-8B · NPU | 0 | 14 | 36 | 50 |
| Llama-3.1-8B · GPU | 0 | 18 | 29 | 47 |
| Qwen2.5-0.5B · NPU | 0 | 1 | 2 | 3 |
| Qwen2.5-0.5B · GPU | 0 | 1 | 2 | 3 |

네 경우 모두 해결한 건수가 0입니다. 8B나 0.5B 모델을 한 번에 답을 받는 방식으로 돌리면
어느 장비에서도 SWE-bench Lite 문제를 풀지 못합니다.

미해결과 적용 실패는 의미가 다릅니다. 미해결은 패치가 적용은 됐지만 테스트를 통과하지 못한 경우이고,
적용 실패는 패치 형식이 깨져서 적용 자체가 안 된 경우입니다. Llama는 적용 실패 비율이 NPU 72%, GPU 62%로,
모델이 형식에 맞는 diff를 잘 만들지 못한다는 점을 보여줍니다.

Qwen은 컨텍스트가 4K로 짧아 oracle 프롬프트(보통 5K~30K 토큰) 대부분이 들어가지 못했고,
그래서 NPU와 GPU 모두 3건만 채점됐습니다.

여기서 중요한 점은 같은 모델의 NPU 결과와 GPU 결과가 거의 일치한다는 것입니다. 정확도는 장비와 무관하다는
뜻이고, NPU로 옮긴다고 정확도가 떨어지지는 않습니다. 정확도를 올리려면 더 큰 모델(32B, 70B)을 쓰거나
에이전트 방식으로 바꿔야 합니다. 참고로 같은 방식(한 번에 답을 받는 방식)이면 GPT-4도 1.7%, Claude 2도 4.8%,
70B 모델도 5.4% 수준입니다.

## 테스트 5: 임베딩 / 리랭커

검색 보조 모델인 Qwen3-Embedding-8B와 Qwen3-Reranker-8B의 처리량도 측정했습니다.
이 항목은 NPU에서만 측정했습니다. GPU 쪽은 임베딩 모델을 아직 켜지 않은 상태라 비교 수치가 없습니다.

| 모델 | 처리량 |
|---|--:|
| Qwen3-Embedding-8B | 1.17건/초 |
| Qwen3-Reranker-8B | 1.17건/초 |

배치 크기를 1, 16, 64로 바꿔도 처리량이 1.17로 고정됐습니다. 배치 처리가 동작하지 않는다는 뜻인데,
prebuilt 모델이 배치 1로 컴파일된 것으로 보입니다. 지금 상태로는 대량 검색에 쓰기 어렵고,
배치를 지원하는 형태로 다시 컴파일하는 것이 후속 과제입니다.

## 어떤 장비를 언제 쓰면 되나

측정 결과를 보면 두 장비의 성격이 분명하게 갈립니다. RNGD는 응답이 빠른 장비이고,
A6000은 한꺼번에 많이 처리하는 장비입니다.

| 상황 | 권장 | 이유 |
|---|---|---|
| 소수 사용자용 코드 도구, IDE 플러그인 (동시 16명 이하) | RNGD NPU | 8B 단일 속도 30% 빠름, 16명까지 처리량도 우세 |
| 응답 지연이 중요한 대화형 서비스 | RNGD NPU | 첫 토큰·토큰 간격 모두 NPU가 짧음 |
| 동시 접속이 많은 공용 API (32명 이상) | A6000 GPU | 32명부터 역전, 128명에서 66% 우세 |
| 1B 미만의 작은 모델 서빙 | A6000 GPU | 작은 모델은 전 구간 GPU 우세 |
| 코드 정확도가 가장 중요한 경우 | 장비보다 모델 선택 문제 | 8B는 둘 다 0%, 32B 이상이나 에이전트 방식 필요 |

8B 모델을 동시 16~32명 이하로 서빙한다면 RNGD가 GPU보다 빠르고 안정적입니다. 동시 사용자가 그보다 많으면
GPU의 처리량이 앞섭니다. 코드 수정 정확도는 두 장비가 같으므로, NPU로 바꾼다고 정확도가 손해 보는 일은 없습니다.

## 남은 과제

32B, 70B 모델은 NPU prebuilt가 RNGD 4카드를 요구해 이번 측정에서 빠졌습니다. 정확도가 중요한 비교는
4카드 환경에서 따로 해야 합니다.

전력과 비용은 이번 측정 범위 밖입니다. 두 장비의 제품군이 달라 전력당, 비용당 성능은 별도 측정이 필요합니다.

임베딩과 리랭커는 GPU 쪽을 측정하지 않아 NPU 수치만 있습니다.

SWE-bench는 Lite 300건 중 50건만 봤습니다. 전체나 Verified로 넓히면 숫자는 달라질 수 있지만,
0%라는 큰 그림은 바뀌지 않을 가능성이 높습니다.
